# techfig Visual Quality Improvement Plan

**Status:** Proposed — not yet implemented.
**Date:** 2026-07-05

## Background / How to pick this up in a fresh session

techfig is a code-first scientific figure toolkit: a JSON spec (elements + connections)
is rendered deterministically to SVG via `techfig/engines/diagrams.py` →
`techfig/utils/svg_builder.py` (drawsvg), linted by `techfig/engines/geo_linter.py`,
optionally critiqued by a vision LLM (`techfig/engines/aesthetic_critic.py`), and
rasterized to PNG by `techfig/utils/export.py` (cairosvg → rsvg-convert → inkscape CLI
fallback chain).

The motivating benchmark is `examples/figures/light_trapping_comparison.md`, which
compares:

- `examples/figures/light_trapping_pc_solar_cell_fig1_techfig.png` — rendered from
  `examples/figures/light_trapping_pc_solar_cell_fig1_spec.json` by this pipeline
- `examples/figures/light_trapping_pc_solar_cell_fig1_venice_gpt_image_2.png` — the
  same figure generated by a diffusion image model (Venice gpt-image-2)

The image-gen result looks journal-quality (title banner, gradient-shaded sphere
lattice, metallic reflector, arrowheads, legend, boxed formula panels, real chart with
axes/ticks) but contains a physics error and is uneditable. The techfig result is
correct and editable but visually plain and has broken label placement. **Goal: keep
techfig's determinism/editability while closing the visual-polish gap.** Re-render the
light-trapping spec after each phase and eyeball it against the Venice PNG — that
comparison is the acceptance test for this whole plan.

The original idea prompting this plan was to build on Inkscape's extensions API
(https://inkscape.gitlab.io/extensions/documentation/) for programmatic layers,
shapes, and shading. **Conclusion after code review: don't** — see the last section.
The gap is caused by three things inside techfig itself, all fixable in pure Python.

## Diagnosis — issues traced to code

### 1. The text-overlap resolver actively destroys good layouts (worst offender)

Phase 2 of `_resolve_text_overlaps` in `techfig/engines/diagrams.py` (~lines 125–134):
any text element whose bbox overlaps **any shape** is moved to
`y = shape_bottom + text_height + 8`.

But text overlapping a shape is usually *intentional*: axis labels next to a chart
box, tick labels, layer names on/near layer rects. In the light-trapping figure this
single behavior dumped "Wavelength (nm)", the 400/600/800 tick labels, "Abs.",
"Active Absorber Layer (a-Si)", the plot title, and the curve legend labels to the
bottom of the canvas — they were correctly placed in the spec. This accounts for
roughly half of the visual damage in the techfig PNG.

The same shove-it-down logic exists in `fix_text_overlaps` in
`techfig/engines/geo_linter.py` (Phase 2 there is text-text only, but the CLI `--fix`
path plus the renderer's resolver compound).

### 2. No styling layer — everything is flat by construction

In `techfig/utils/svg_builder.py`, every shape (`add_box`, `add_circle`, …) is drawn
with `stroke=fill` (same color) and default `fill_opacity=0.15`
(`DEFAULT_FILL_OPACITY`). Consequences:

- No outline contrast, no depth, no material identity (the Venice reflector reads as
  brushed metal; techfig's is a gray rect at 20% opacity).
- No gradients, no drop shadows, no patterns — the builder has no `<defs>` support
  at all.
- No typography hierarchy: one `font_size` in the style config; no bold titles, no
  named text roles. The techfig figure has **no title**.
- `add_line` has no arrowhead option — arrowheads only exist on element-to-element
  `add_arrow` connections, so free light rays are bare lines.

### 3. The primitive vocabulary is too low-level

Supported element types are only box/circle/diamond/ellipse/triangle/text/line (plus
registered components). The spec-authoring LLM therefore fakes everything:

- The inset absorption chart is **18 hand-placed line segments** plus loose text for
  axes/ticks/legend (`planar_curve1..9`, `pc_curve1..9` in the spec JSON) — even
  though techfig already ships a matplotlib engine (`techfig/engines/figures.py`).
- The photonic-crystal lattice is 6 lonely circles (`ph1..ph6`); the Venice image has
  a dense shaded 5×6 sphere array.
- The "Light Trapping Mechanism" sidebar is **9 separate text elements** manually
  line-broken, and the formula block is 5 more. Every extra element is another chance
  for the LLM to misplace something and for the resolver/linter to mangle it.
- Formulas are plain strings ("4n² · d", "λ_peak") — no real math typesetting.

### 4. Text bounding boxes are guesses

`len(text) * font_size * 0.6` is used for text extents in `diagrams.py`
(`_estimate_text_bbox`), `geo_linter.py` (`_text_bbox` and the lint bbox pass), and
`svg_builder.py` (`add_text` element registration). Routinely off by 30%+, so overlap
detection both false-positives (triggering the destructive resolver) and misses real
collisions.

### 5. The PNG rasterizer will drop any styling we add

`techfig/utils/export.py` `_svg_to_raster` tries **cairosvg first**. cairosvg does not
support SVG filters (drop shadows, blur) and has patchy gradient/`tspan` support. A
correct Chromium-based renderer already exists in the codebase:
`aesthetic_critic.render_to_png` (Playwright). Any Phase-2 styling work is wasted
until the default PNG path renders it.

### 6. The critique loop exists but isn't wired

`geo_linter.py` (deterministic geometry score) and `aesthetic_critic.py` (vision-LLM
score + actionable feedback, optional, needs API key) exist but nothing iterates:
render → critique → revise spec → re-render. Diffusion models look good partly
because they implicitly run many "does this look right" corrections; techfig does
zero.

Also relevant: `techfig/engines/pretty.py` currently calls `litellm.image_generation`
with a generic prompt and **never actually passes the rendered base image** (the
comment block admits this) — so its output is unrelated to the figure. Either wire it
as image-to-image for style-reference use or treat it as dead weight.

## Plan

### Phase 1 — Stop destroying good layouts (small; do first)

1. **Fix the overlap resolver** (`diagrams.py` `_resolve_text_overlaps` Phase 2, and
   the analogous logic in `geo_linter.fix_text_overlaps`):
   - Text-over-shape overlap is not an error by default. Only displace text on
     **text-text** collision.
   - When displacing, move toward the nearest clear space (try up/down/left/right of
     the current position), not unconditionally below the shape.
   - Add an opt-out: elements with `"pin": true` are never moved.
2. **Real font metrics**: replace the `len*0.6` heuristic with measured extents
   (`PIL.ImageFont.getbbox` with a bundled/system font, or matplotlib
   `TextPath`/`FontProperties` since matplotlib is already a dependency). Apply in
   `diagrams.py`, `geo_linter.py`, and `svg_builder.py` consistently — one shared
   helper in `techfig/utils/`.
3. **Rasterizer order**: make the Playwright/Chromium renderer (reuse
   `aesthetic_critic.render_to_png`, moved into `utils/export.py` or imported) the
   preferred PNG backend when available; fall back to rsvg-convert / inkscape CLI /
   cairosvg. Alternatively add `resvg` (fast, correct, small) as preferred backend.

### Phase 2 — Styling layer in SVGBuilder (biggest visual payoff)

1. **`<defs>` support**: named linear/radial gradients, a soft drop-shadow filter
   (`feDropShadow` or gaussian-blur composite), hatch/dot patterns.
2. **Material presets** resolvable from the spec's `color`/`fill` field: e.g.
   `metal` (vertical gray gradient, brushed look), `semiconductor` (deep
   purple/indigo gradient), `glass` (pale blue, high transparency, white top
   highlight), `dielectric`, `substrate`. A spec then says `"fill": "metal"`.
3. **Stroke ≠ fill**: default outline = fill darkened ~30%, raise default fill
   opacity for material shapes (keep light fills only for annotation panels).
4. **Typography roles**: named styles (`title`, `subtitle`, `label`, `annotation`,
   `tick`) mapping to size/weight/color; text elements reference a role instead of
   raw `font_size`. Add a top-level `"title"` (+ optional `"subtitle"`) field to the
   spec schema that renders a bold centered header.
5. **Arrows and paths as first-class**: `"marker": "end"|"start"|"both"` on `line`
   elements; new `path`/`polyline` element type with optional smoothing (Catmull-Rom
   → cubic Béziers) so one light ray with reflections is one element with an
   arrowhead, not 3–4 bare lines.

### Phase 3 — Mid-level components (close the richness gap)

New element types in `diagrams.py` (this is where the "programmatic functions that
draw complicated shapes/shading" idea belongs — inside the engine, not in Inkscape):

1. **`textblock`** — multi-line wrapped text (width + anchor + role) with optional
   rounded-rect panel background and padding. Replaces 9-element paragraphs and
   removes most overlap problems at the source.
2. **`plot`** — inline chart: takes data series + axis labels, renders via the
   existing matplotlib engine to SVG, embeds as a positioned group. Real axes, ticks,
   smooth curves, legend for free. Kills the 18-segment faked chart.
3. **`lattice`** — n×m (or hex) array of a base shape with a shared radial-gradient
   sphere style; parameters: rows, cols, spacing, r, style. One element for the
   photonic-crystal layer.
4. **`callout`** — label + leader line anchored to a target element:
   `{"ref": "pc_layer", "side": "right", "offset": 12}`. Anchored labels are how the
   Venice image gets its clean aligned label columns, and they stay correct when the
   target moves.
5. **`legend`** — list of (swatch/line-style, label) rows in a bordered panel.
6. **Math text** — render formula strings through matplotlib mathtext to SVG paths
   (real subscripts, Greek, fractions). Either a `math` element type or automatic
   `$...$` detection in text.

Each new component must also be documented in `SKILL.md` / the sketch-interpreter
prompt (`techfig/engines/sketch_interpreter.py` `SKETCH_PROMPT`) — the spec-authoring
LLM can only use vocabulary it is told about, and richer vocabulary = fewer elements
to misplace.

### Phase 4 — Close the aesthetic loop

1. **`--polish N` CLI mode**: render PNG → `aesthetic_critic.score_aesthetic` →
   feed feedback + current spec to an LLM to produce a revised spec → geo-lint
   gatekeeper → re-render. 2–3 iterations. Stop early when score plateaus ≥ ~0.8.
2. **Style-reference mode** (the comparison doc's own conclusion): optionally pass a
   diffusion-generated image of the same figure to the vision critic as *layout/style
   inspiration only* while revising the spec — techfig keeps ownership of every
   coordinate and label so correctness is preserved.
3. Fix or delete `pretty.py` (currently generates an image unrelated to the figure —
   see Diagnosis #6).

### Suggested sequencing

Phase 1 + items 1–4 of Phase 2 ≈ 1–2 days and transform the benchmark figure.
Then `plot` and `textblock` (Phase 3) are the next highest leverage. Phase 4 last.

**Acceptance test:** re-render `examples/figures/light_trapping_pc_solar_cell_fig1_spec.json`
(update it to use new vocabulary as phases land) and compare against
`light_trapping_pc_solar_cell_fig1_venice_gpt_image_2.png`; update
`light_trapping_comparison.md` with the new result.

## Why not the Inkscape extensions API

The extensions API (`inkex`) is designed to run inside a GUI Inkscape session
operating on a live document. Driving it programmatically means shelling out to a
full Inkscape process per operation (~1–2 s startup each), a heavyweight install
dependency, and adopting Inkscape's document model. Everything it would provide has a
lighter pure-Python answer:

| What Inkscape extensions would give | Better option for techfig |
|---|---|
| Gradients, shading, layers | Plain SVG `<defs>` + groups in SVGBuilder (Phase 2) |
| Accurate text extents | PIL / matplotlib font metrics (Phase 1) |
| Path booleans / offsets / generative shapes | `svgpathtools` + `shapely`, if ever needed |
| Rendering fidelity | Playwright/Chromium (already in repo) or `resvg` |

Where Inkscape stays valuable, unchanged: as a fallback CLI rasterizer in
`export.py`, and as the end user's manual editor for the SVG output — which is the
product's core selling point.
